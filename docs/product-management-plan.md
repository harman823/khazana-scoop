# Cross-Repo Product Management Plan

## Goal

Build a real product-management system across both repositories:

- Dashboard repo: `/Users/yankitrajor/Desktop/Apoorav/Personal/Khazana-Scoop`
- Storefront repo: `/Users/yankitrajor/Desktop/Apoorav/Personal/khazana-scoop-web`

The dashboard must become the place where the team manages products, categories, collections, images, discounts, and stock. The storefront must stop using hard-coded catalog data and instead read the same Supabase-backed catalog that the dashboard writes.

This plan replaces the earlier version because the current codebases do not match that original assumption:

- The dashboard already has a working Supabase inventory and order system.
- The storefront still uses static product content plus a separate Prisma-backed mystery-scoop flow.
- The two repos do not yet share one catalog model.

## Source of Truth Decision

**Supabase in the dashboard repo will be the source of truth for the public catalog.**

That means:

- Product records will live in the dashboard Supabase schema.
- Product edits will happen in the dashboard app.
- The storefront will fetch those products for product listing, product detail, homepage category sections, and collection sections.
- Static `featuredProducts` in the storefront will be removed or reduced to fallback/demo use only.

## What Exists Today

### Dashboard repo today

- Has a Supabase-backed `products` table with stock and cost tracking.
- Has `orders`, `order_items`, `stock_movements`, `expenses`, and `scoop_types`.
- Has working pages for stock, orders, expenses, dashboard, and query testing.
- Uses lightweight cookie auth for owner access.

### Storefront repo today

- Has a public UI and several designed pages.
- Uses static `featuredProducts` for catalog pages and homepage sections.
- Uses Prisma for mystery scoop tiers, add-ons, account/login, checkout, and admin fulfillment pages.
- Does not read catalog products from the dashboard Supabase schema.

## Product Scope

We will deliver these capabilities:

1. Dashboard product CRUD
2. Category CRUD
3. Collection CRUD
4. Product image management
5. Product colour management
6. Product and category/collection discount management
7. Stock movement tracking
8. Storefront sync with the same catalog data
9. Standard product detail pages on the storefront
10. Automatic stock deduction for standard catalog orders

## Guiding Implementation Rules

1. Do not create a second catalog source of truth.
2. Do not break the existing mystery-scoop checkout flow while catalog work is landing.
3. Extend the existing dashboard Supabase schema instead of inventing a parallel Prisma product catalog in the storefront.
4. Move the storefront from static catalog rendering to Supabase-backed rendering in phases.
5. Keep the current dashboard order and stock system working while we expand it.

## Phase Plan

### Phase 1: Data Model Alignment

Extend the existing dashboard Supabase schema instead of replacing it.

#### Existing table to extend

`products`

Add fields:

- `slug text unique not null`
- `description text`
- `base_price numeric(12,2) not null default 0`
- `active boolean not null default true`
- `primary_image_url text`
- `available_colours text[] not null default '{}'`
- `sort_order integer not null default 0`
- `category_id bigint references categories(id) on delete set null`

#### New tables

`categories`

- `id bigint primary key generated by default as identity`
- `name text not null unique`
- `slug text not null unique`
- `active boolean not null default true`
- `sort_order integer not null default 0`
- `created_at timestamptz default now()`

`collections`

- `id bigint primary key generated by default as identity`
- `name text not null unique`
- `slug text not null unique`
- `active boolean not null default true`
- `sort_order integer not null default 0`
- `created_at timestamptz default now()`

`product_collections`

- `product_id bigint references products(id) on delete cascade`
- `collection_id bigint references collections(id) on delete cascade`
- `primary key (product_id, collection_id)`

`product_images`

- `id bigint primary key generated by default as identity`
- `product_id bigint references products(id) on delete cascade`
- `url text not null`
- `alt_text text`
- `sort_order integer not null default 0`
- `created_at timestamptz default now()`

`discounts`

- `id bigint primary key generated by default as identity`
- `target_type text not null check (target_type in ('product','category','collection'))`
- `target_id bigint not null`
- `amount numeric not null`
- `type text not null check (type in ('fixed','percent'))`
- `start_at timestamptz`
- `end_at timestamptz`
- `active boolean not null default true`
- `created_at timestamptz default now()`

#### Keep existing tables

- `stock_movements`
- `orders`
- `order_items`
- `scoop_types`
- `expenses`

These stay in place and will be extended only if needed.

### Phase 4: Storefront Catalog Integration

Replace static catalog rendering in `khazana-scoop-web` with Supabase catalog queries.

#### Remove or demote static catalog data

- `src/lib/products.ts` should no longer be the primary catalog source.
- Homepage category and collection sections should stop using static arrays.
- Product detail pages should stop being generated from hard-coded slugs.

#### Storefront features to implement

- Dynamic product listing from Supabase
- Dynamic product detail by slug
- Dynamic homepage category sections
- Dynamic explore-collections section
- Discount badge and crossed-out original price
- Product gallery using `product_images`
- Colour swatches or text chips using `available_colours`

#### Storefront data access

Add a Supabase catalog client or query layer for:

- `getCatalogProducts()`
- `getCatalogProductBySlug(slug)`
- `getHomepageCategoriesWithProducts()`
- `getCollectionsWithProducts()`
- `getActiveDiscounts()`

### Phase 5: Order and Stock Deduction Strategy

This needs to be done carefully because the two repos currently have different order systems.

#### v1 rule

Standard catalog products on the storefront should write to the dashboard-backed `orders` and `order_items` tables so stock deductions happen against the same `products` table.

#### Existing mystery scoop flow

The current mystery-scoop custom flow in the storefront can remain on its current Prisma path temporarily.

That means we will have:

- standard catalog orders flowing through the dashboard Supabase catalog tables
- mystery scoop custom orders still using the current mystery-scoop order flow until migrated later

#### Automatic stock deduction

For standard catalog orders:

- create the order
- insert `order_items`
- decrement `products.stock_quantity`
- write a `stock_movements` row with reason like `Order #123`

Implementation choice:

- prefer a database trigger or RPC for atomicity
- fall back to a server action/transaction-like sequence only if required

### Phase 6: Visual and Content Sync

Once catalog data is connected, keep the current design work in the storefront but switch the content source.

This includes:

- homepage sections reading live products
- product pages reading live descriptions, images, prices, colours
- collection sections reading live collection assignments
- category sections reading live category assignments

## Risks and Notes

1. The storefront currently has a separate mystery-scoop order architecture. We should not merge that blindly into the catalog work.
2. The dashboard already has working stock and order behavior, so schema changes should extend it rather than rewrite it.
3. Writing to the dashboard repo from this Codex workspace requires permission because it is outside the current writable root.
4. The current storefront design work can stay, but the underlying content source must change from static data to Supabase.

1. Extend dashboard Supabase schema.

4. Switch storefront products page and product detail page to Supabase.
5. Switch homepage category and collection sections to Supabase.
